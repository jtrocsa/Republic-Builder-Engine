## 1. Teacher-facing UI and accessibility — see `TEACHER-UI-ACCESSIBILITY-AUDIT.md`

Headline finding: Chronicle already has **better native-accessible coverage than a from-scratch audit would assume** — `aria-live="polite"` + `role="status"` appears in 17 places (quest feedback, dialogue bubbles, challenge-completion messages), `aria-expanded` is used correctly on both the Manage Content unit accordion (`main.js:2684`) and a "Show/Hide Full Text" toggle (`main.js:2269`), and the password field already has a working Show/Hide toggle with `aria-pressed` (`main.js:1990-1992`). The real gaps are narrower and more specific: **no `Escape`-key handling exists anywhere in `main.js`** (only movement `keydown`/`keyup` at `main.js:8000-8001`), and the one confirmation dialog in the app (teacher-added-question delete, `main.js:2754`) is a plain two-button pair with no focus trap, no `Escape`-to-cancel, and no native `<dialog>`.

## 2. Screen and workflow state — see `WORKFLOW-STATE-AUDIT.md`

Headline finding: the state machine is a flat 34-entry `VALID_SCREENS` Set (`main.js:1451`) behind a single `render()` switch, with a resilient top-level try/catch fallback to `instituteScreen()` on any render error, and a boot-time guard (`main.js:1487-1491`) that redirects stale/volatile/invalid `currentScreen` values back to a safe screen. `CLICK_HANDLER_GROUPS` (`main.js:7522`, 12 handler functions) is already a clean first-match-wins dispatcher — this is the Phase 13 decomposition of what used to be one 570-line function, and it still holds up. **No forcing function exists for a formal state-machine library** (XState or similar): the current pattern already has (a) an explicit valid-state allowlist, (b) a redirect-on-invalid-state guard, and (c) a render-error recovery fallback — the three things a state-machine library would mainly buy. The real opportunity is narrower: `previewSession` (`main.js:1610`) is a second, ad hoc piece of screen-adjacent state that lives outside `progress`/`VALID_SCREENS` entirely and is guarded manually at each of its ~6 call sites rather than through one policy.

## 3. Quest-type metadata and orchestration duplication

This is the sharpest, most concrete finding in the whole audit, confirmed by direct code read rather than inferred from documentation (no existing doc — including `QUEST-TYPE-ARCHITECTURE.md`, read in full for this pass — mentions it).

**There are two independent, non-reusing status/hint-derivation systems for the same four quest types:**

1. **`isChallengeQuestComplete` / `challengeQuestAnsweredAny` / `challengeQuestPartialSuccess` / `challengeQuestHint`** (`main.js:1819-1848`) — four small shared helper functions, each branching on `questType` internally. Used by `archiveChallengesScreen()` and `investigationScreen()`.
2. **`practiceCheckScreen()`** (`main.js:5525-5609`) — four separate, fully inline blocks (one per quest type: MCQ at 5528-5546, Sequencing at 5548-5563, Evidence Organizing at 5565-5583, HIPP at 5585-5607), each independently re-deriving its own `status` string and feedback HTML directly from `gradeQuest()`'s result — **without calling any of the four helpers in (1)**.

The two systems don't even agree on vocabulary: system (2)'s HIPP block has a `"partial"` status (`result.pointsEarned > 0` but not complete) that system (1) has no equivalent for — `challengeQuestHint()`'s HIPP branch only distinguishes "answered" from "not." A future change to how HIPP partial credit is described has to be made in two places by hand, with no test or lint rule that would catch a drift between them.

**Root cause:** `QUEST_TYPES` (`apps/web/src/quest-types/index.js`) entries are exactly `{schema, listSchema, render, grade}` — confirmed by direct read of `index.js` and all four modules (`generic/mcq-quest.js`, `generic/sequencing-quest.js`, `history/evidence-organizing-quest.js`, `history/source-analysis-quest.js`). `grade()`'s return shape differs per type by design (`{answered, correct}` vs. `{allPlacedCorrectly, reflectionOk, complete}` vs. `{results, pointsEarned, pointsPossible, complete}`), so every consumer of `gradeQuest()` is forced to branch on `questType` itself to get a uniform status out of it. `main.js` did this twice, independently, at two different times (Phase 11 for Practice Check, the second-wave pass for Archive/Investigation Challenges) — see `FOCUSED-MODERNIZATION-ROADMAP.md` for the consolidation recommendation.

## 4. Authoring and content-management infrastructure — see `AUTHORING-SYSTEMS-AUDIT.md`

Headline finding: the custom-content-authoring pipeline (`apps/web/src/engine/custom-content-authoring.js`, Phase 26) is a real, Zod-validated, schema-driven form-to-content converter — not a naive text-blob store. The known gaps are exactly the ones the decision log already self-reported and none have been quietly fixed since: the curated-alternate picker UI was never built despite the backend supporting it since Phase 23 (`contentUiState.slots[].alternatives` is computed and unused); "+ Add new question" is deliberately disabled rather than fixed, because the render path (`archiveChallengesScreen()`/`practiceCheckScreen()`) only iterates official quest ID lists and can never discover a `mode: 'addition'` row; SAQ has no quest type at all, so it can't appear in the picker even disabled-vs-hidden is a design choice already made correctly (shown, disabled, with a tooltip, not silently omitted).

## 5. Playwright and regression testing — see `PLAYWRIGHT-ADOPTION-PLAN.md`

Headline finding, confirmed by `Glob` (no `tests/e2e/`, no `playwright.config.*` anywhere in the repo) and by reading all four Playwright mentions across `POST-MINIMAL-ARCHITECTURE-REASSESSMENT.md`/`QUEST-TYPE-ARCHITECTURE.md`: Playwright has been used in **every one of 20+ documented phases** as an ad hoc scripted verification pass run manually during a session, and has **zero committed, repeatable test artifacts** today. This is a real, unusually strong evidence base for adopting a small permanent suite — see the companion doc for the proposed scope.

## 6. Supabase UI and data-operation behavior

Audited `apps/web/src/repositories/` in full (14 files, 36–362 lines each) plus direct reads of `progress-repository.js`, `remote-classroom-repository.js`, and `remote-submission-repository.js`.

**The pattern is more consistent than a from-scratch guess would assume.** Nearly every repository function follows `const { data, error } = await supabase...; if (error) throw error;`, and every screen that calls one wraps the call in `try { ... } catch (err) { xUiState.error = err.message || "<fallback>"; } render();` — confirmed identically shaped in `loadTeacherDashboardData()` (`main.js:2405-2422`), `openGradingScreen()` (`main.js:2440-2457`), and the auth screens' `authUiState.error`/`authUiState.pending` handling (`main.js:1990-2036`). Every one of those screens also renders the same `${xUiState.error ? '<p class="feedback error">...' : ""}` markup fragment. This is **duplicated boilerplate, not a broken pattern** — five near-identical try/catch-then-render blocks and five near-identical error-markup fragments, none of them actually buggy.

The two deliberate exceptions — `progress-repository.js`'s background sync (`main.js`'s debounced `saveProgress` push) and `remote-submission-repository.js`'s `recordSubmission()` — intentionally swallow errors via `console.error()` and return `null` rather than throwing. This is **correct, not a gap**: both are fire-and-forget background writes (progress autosync, AI-feedback logging) where surfacing a network hiccup mid-gameplay would be actively disruptive to a student. Don't "fix" this into a throwing pattern.

**No loading-spinner/skeleton pattern exists** — screens either render immediately with cached state or show a static "Loading submission…" string (`gradingScreen()`, `main.js:2375`) while an async loader runs in the background and calls `render()` again on completion. This is adequate for the current data volumes (one classroom, one submission) and not a gap worth a library for.

**Conflict resolution**: `resolveProgressConflict()` (`progress-repository.js:31-36`) is a small, well-commented last-write-wins-by-timestamp function — already minimal and correct for its one real use (rotating school Chromebooks), no generalized offline/optimistic-sync library would reduce its ~6 lines.

## 7. Input and accessible interaction

Confirmed via direct grep of `main.js`: two `keydown`/`keyup` listeners (`main.js:8000-8001`) drive field/hub movement only — no Escape, no other global keyboard shortcut. Drag-and-drop uses four native HTML5 DnD listeners (`dragstart`/`dragover`/`dragleave`/`drop`, `main.js:7993-7996`) shared across `data-map-piece`, `data-cargo-card`, `data-sequence-item`, `data-evidence-slot`, and related attributes. **Keyboard alternatives to every drag-and-drop interaction already exist** (Phase 12): sequence ↑/↓ move buttons and an evidence "place in" `<select>`, both wired through shared `applySequenceOrder`/`applySequenceMove`/`applyEvidencePlacement` helpers reused by the drop handlers themselves — this is a genuinely solid existing pattern, not a gap. No touch (`touchstart`) or gamepad support exists anywhere — confirmed absent, and per the prompt's own rule, absent a concrete current need (remapping, touch, gamepad, or a repeated input bug), this stays **Defer**, not a library candidate.

## 8. Audio

`apps/web/src/engine/audio-engine.js` (207 lines) is self-contained: owns its own `AudioContext`, a single master gain node, and `audioEnabled` localStorage persistence, with a doc-commented scope boundary at the top of the file explicitly separating "data tables" (note frequencies, per-scene music sequences) from logic that stays out of the engine (`sceneForMusic()` deliberately lives in `main.js` because it reads screen-routing state). Procedural oscillator/noise synthesis only, no sample assets. This is small and adequate by the audit's own stated bar ("don't recommend replacing it if small, isolated, and adequate") — **Keep as-is**.

## 9. Remaining `main.js` concentration

At 8,005 lines, the largest identifiable blocks by function are: the Manage Content case editor (`manageContentCaseScreen()` and its ~15 supporting markup functions, roughly `main.js:2600-3230`, source-grouped collapsible authoring UI); the teacher dashboard tab bodies (`teacherClassroomsTabMarkup`/`teacherAssignmentsTabMarkup`/`teacherSourcesTabMarkup`/`teacherUnitsTabMarkup`, referenced at `main.js:2174-2180`); and the four-quest-type Practice Check screen (`main.js:5525-5609`, the duplication site from §3 above). None of these are movement/collision/camera/NPC code — they're all screen-assembly and template-literal HTML, matching the project's own stated view that only that category is off-limits for extraction (per `CLAUDE.md`'s "do not physically extract" rule, which is scoped to movement/collision/camera/NPC specifically, not all of `main.js`). No extraction is recommended in this audit purely on line-count grounds, per the prompt's own rule — see the decision table below for the two extractions that do meet the bar (a cohesive boundary + current callers + concrete regression-risk accounting), both narrowly scoped to §3's duplication, not a general `main.js` breakup.
